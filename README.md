# Deploy Gitea in a Custom VPC with Multiple EC2 Instances
This project represents the full deployment of a secure cloud architecture on AWS, where I built a public reverse proxy and two private backend services inside a properly segmented VPC. The goal was to design an environment that separates public and private traffic, enforces least‑privilege access, and demonstrates that each component behaves exactly as intended. Throughout the process, I configured the networking, deployed the services, validated the routing, and collected evidence showing that everything works securely and consistently.

The architecture is based on a single VPC with a CIDR block of 10.0.0.0/16. Inside this VPC, I created a public subnet (10.0.1.0/24) and a private subnet (10.0.2.0/24). The public subnet contains the nginx reverse proxy and the NAT Gateway, while the private subnet contains the Gitea backend and the Audit/Event API backend. The public subnet routes outbound traffic through the Internet Gateway, and the private subnet routes outbound traffic through the NAT Gateway. This ensures that only the reverse proxy is publicly reachable, while the backend services remain fully isolated.

To support this architecture, I configured two route tables. The public route table sends all internet‑bound traffic to the Internet Gateway, allowing the nginx instance to be accessible from outside the VPC. The private route table sends outbound traffic to the NAT Gateway, which allows the backend instances to install packages and update software without ever being exposed to the public internet. This routing setup is essential for maintaining the separation between public and private resources.

Security groups reinforce this separation. The nginx security group allows HTTP traffic from anywhere and SSH only from my IP address. The backend security group is much more restrictive: it only allows port 3000 from the nginx security group for Gitea, port 5000 from the nginx security group for the API, and SSH from my IP. No public access is allowed to either backend service. This ensures that the only way to reach Gitea or the API is through the reverse proxy.

All three EC2 instances—the nginx reverse proxy, the Gitea backend, and the Audit/Event API backend—are running successfully, and I included screenshots from the AWS console to show their status. Once the services were deployed, I verified that Gitea works through the reverse proxy by curling the public IP of the nginx instance. The response returned the full Gitea HTML homepage, which confirms that nginx is correctly forwarding traffic into the private subnet. I repeated the same process for the API by curling the /api/ path on the public IP, and the response confirmed that the API is reachable only through nginx.

To demonstrate that the backend services are not publicly accessible, I attempted to reach both ports 3000 and 5000 directly from my local machine. Both attempts failed, which is exactly what should happen in a secure architecture. These failures confirm that the backend services are properly isolated and that the security groups and subnet design are working as intended.

Finally, I verified that API requests are being logged correctly. By checking the log file inside the API instance, I could see each request coming through nginx, including timestamps and request paths. This confirms that the API is running with logging enabled and that the reverse proxy is forwarding traffic as expected.

Overall, this project demonstrates a complete and secure cloud architecture with proper network segmentation, a public reverse proxy, private backend services, correct routing and NAT behavior, enforced security boundaries, and verified connectivity and logging. All required screenshots and the architecture diagram are included in the repository.
